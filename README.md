---

## 2. Документація по розробленому веб-застосунку

### Призначення застосунку
**Task Tracker** — це простий веб-сервіс для створення та відстеження задач.  
Кожна задача містить:
- унікальний ідентифікатор (**id**)
- назву (**title**)
- статус (**pending/done**)
- час створення (**created_at**)

---

### Налаштування середовища

Для локального запуску застосунку потрібно мати встановлений **Python 3** та **MariaDB**:

1. Клонувати репозиторій  
2. Створити віртуальне середовище:  
   `python3 -m venv venv`  
   `source venv/bin/activate`  

3. Встановити залежності:  
   `pip install -r requirements.txt`  

4. Створити базу даних у MariaDB та виконати міграцію:  
   `python3 migrate.py`  

---

### Запуск застосунку

Оскільки застосунок використовує аргументи командного рядка:

```bash
python3 app.py --port=5000 --db-user=taskuser --db-pass=taskpass --db-name=taskdb
```

---

### Документація по API-ендпоінтах

#### 1. Кореневий ендпоінт
- `GET /` — повертає HTML-сторінку зі списком ендпоінтів бізнес-логіки

#### 2. Бізнес-логіка
- `GET /tasks` — отримати всі задачі  
- `POST /tasks` *(параметр: title)* — створити нову задачу  
- `POST /tasks/<id>/done` — позначити задачу як виконану  

#### 3. Службові ендпоінти (Health Checks)
- `GET /health/alive` — завжди повертає **200 OK**  
- `GET /health/ready` —  
  - **200 OK** (якщо є зв'язок з БД)  
  - **500 Internal Server Error** (якщо БД недоступна)

## 3. Документація по розгортанню

### Віртуальна машина

1. **Базовий образ**  
   Ubuntu Server 22.04 LTS (або 24.04 LTS)

2. **Вимоги до ресурсів**
   - CPU: 1 Core  
   - RAM: 1–2 GB  
   - Disk: 10+ GB  
   - Мережа: NAT + Port Forwarding (SSH)

3. **Спеціальні налаштування OS**
   - Стандартна розбивка диска  
   - Створено адміністратора під час інсталяції  

4. **Доступ (Credentials)**
   - Через Console (VirtualBox) або SSH  
   - Після запуску скрипта:
     - основний користувач блокується  
     - використовуються користувачі `student` та `teacher`

---

### Запуск автоматизації розгортання

Скрипт `deploy.sh` автоматизує:
- встановлення пакетів  
- створення користувачів  
- налаштування Nginx  
- MariaDB  
- Systemd  

```bash
chmod +x deploy.sh
sudo ./deploy.sh
```

---

### Інструкція з тестування розгорнутої системи

#### 1. Перевірка Nginx та API (порт 80)

```bash
curl -H "Accept: application/json" http://127.0.0.1/tasks
```
*Повертає: дані*

```bash
curl -I http://127.0.0.1/health/alive
```

*Повертає: 403 Forbidden через Nginx*

---

#### 2. Перевірка Health Checks (порт 5000)

```bash
curl -i http://127.0.0.1:5000/health/ready
```

- Повертає **200 OK**

При вимкненні БД:
```bash
systemctl mask mariadb
```

- Повертає **500 Internal Server Error**

---

#### 3. Перевірка прав користувачів

- `student` — Адміністративні права, або можливість ескалації до root 
- `app` — Мінімально необхідні права  
- `teacher` — Адміністративні права, або можливість ескалації до root, при першому вході вимагається зміна пароля  
- `operator` — Обмежений доступ через sudo, при першому вході вимагається зміна пароля:
  - запуск веб-застосунку
  - зупинка веб-застосунку
  - перезапуск веб-застосунку
  - перегляд статусу веб-застосунку
  - Перезавантаження конфігурації (reload) nginx

---

#### 4. Перевірка Socket Activation

- При зупиненому `mywebapp.service`  
- Запит на порт **5000** автоматично запускає сервіс через `mywebapp.socket`

---

#### 5. Перевірка файлу

```bash
sudo cat /home/student/gradebook
```

- Має містити правильний варіант: **N = 4**

#### 6. Перевірка функціоналу

- Створення задачі:
```bash
curl -X POST -d "title=Cook dinner" http://127.0.0.1/tasks
```

- Оновлення статусу задачі:
```bash
curl -X POST http://127.0.0.1/tasks/1/done
```
